Development Workflow
- bug
	: launchpad / storyboard에서 확인가능
	: 4가지 핵심 tasks
		- 새로운 버그 확정 : new → confirm
		- 일관성 해결 : In progress
		- incomplete bugs 리뷰 : 정보를 더 모아라
		- In-process 버그들 리뷰 : 작업 중 아니면 다시 되돌리기
	: 버그 작업 후 커밋 메시지에 '특정 키워드'를 적어야함

- specificatoins and blueprints (설계 제안 및 추적 절차)
	- specs : 설계도(layout)
	- blueprints : 설계의 실행상태를 보여줌

- Starting a change : 최신상태시작
	- git remote update
	- git checkout master
	- git pull --ff-only origin master

- Git branch names
	- 새기능 : bp/abc, 버그 : bug/123 선호

- Committing changes
	- 새로운 기능(blueprint), 버그일 때 특정 메시지
	- change-Id : 보통 git review -s 쓰면 해결 (안되면 git commit --amend)
	- master or main 브랜치에서 작업 X >> 항상 별도의 작업용 브렌치 ㄱㄱ

- Using signed-off-by : 저작권관련 이슈

## 중요한거 같은데 와닿지가 않음,,

- Adding a Dependency : 아직 승인되지않은 커밋을 내가 작업하고 싶을 때
	- 이미 있는 change(review)에서 작업
	- 이미 있는 parent change가 바뀌어도 그대로 작업 가능 (이러면 내 작업은 child change)
	- git review는 rebase과정이 포함, 안하고 싶으면 추가 옵션 (근데 결국 최종적으론 해야함)

- Rebasing a commit : 서버의 target branch와 내 코드 사이에 충돌 발생 시 rebase(=기준점 맞추기)
	- 하는 방법은 알아서

- Cross-Repository Dependencies
	: 서로 다른 repository에 있는 각각의 change들이 서로 영향
	- manual setup (코드가 합쳐지기 전)
		: work in progress 상태로 만들기
	- CRD (다른 저장소의 코드에 의존)
		: commit msg에 표기 (Depends-On: <Gerrit 리뷰 주소>)
